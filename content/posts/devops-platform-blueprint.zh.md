# 通用 DevOps 平台蓝图

DevOps 平台的目标，是让“正确的路径”比各种临时捷径更容易走。它不只是 CI/CD 脚本、Kubernetes 配置、监控面板和权限策略的集合，而是一个面向内部工程团队的产品：把工程标准沉淀成可复用的工作流，让应用团队在不重复踩坑的前提下安全交付。

这篇文章整理一套通用的平台蓝图。它不绑定具体工具，因为工具会变化，但平台要承担的职责、边界和反馈机制在大多数组织里都是相通的。

配套示例仓库可以参考 [sven0219/devops-platform-blueprint](https://github.com/sven0219/devops-platform-blueprint)，其中包含与这套蓝图对应的示例结构和参考资料。

## 1）先定义平台原则

在选型之前，先明确平台的设计原则：

1. **默认自助化**：团队可以自行创建环境、部署服务、查看日志、申请标准依赖，而不是每一步都等待平台同学处理。
2. **用护栏替代关卡**：安全、成本和可靠性要求应内置在模板和流水线中，而不是发布前才靠人工审核补救。
3. **一切皆代码**：基础设施、策略、部署配置和可观测性规则都应该版本化、可审查、可回滚。
4. **黄金路径不是牢笼**：为大多数场景提供推荐模式，同时为确有必要的特殊场景保留例外机制。
5. **服务团队承担运营责任**：谁拥有服务，谁就要负责它的健康、成本和生命周期；平台要让这种责任变得可执行。

这些原则能帮助平台聚焦真正的目标：更快交付、更少生产事故、更清晰的责任边界，以及更低的认知负担。

## 2）拆分核心层次

一个实用的 DevOps 平台通常包含五个层次。

### 源码与协作层

这一层包括 Git 仓库、分支策略、Pull Request 检查、代码负责人、密钥扫描、依赖审查和发布说明。目标是让每一次变更都能从需求追溯到生产环境。

建议具备：

- 面向常见服务类型的仓库模板。
- 测试、格式化、安全扫描和构建验证等必需检查。
- 服务归属元数据，让评审、告警和事故能路由到正确团队。
- 在适合的场景中自动生成变更日志或发布说明。

### CI/CD 与发布自动化层

流水线层负责把代码稳定地变成可部署制品，并让制品在环境之间有序流转。流水线越“无聊”，越说明它可靠、可预测、可观测。

常见能力包括：

- 一次构建，多环境提升同一个制品。
- 清晰拆分构建、测试、扫描、打包、部署和验证阶段。
- 环境差异配置与应用代码解耦。
- 对关键服务支持回滚、金丝雀或蓝绿发布。
- 部署审计链路能关联提交、制品、审批人和运行版本。

### 基础设施与运行时层

基础设施层提供计算、网络、存储、身份、入口、证书和运行时策略。Kubernetes 很常见，但这套蓝图同样适用于 Serverless 或虚拟机平台。

推荐的基础能力包括：

- 标准环境的 Infrastructure as Code 模块。
- 包含入口、DNS、TLS、自动扩缩容和策略执行的运行时基线。
- 按团队和环境划分命名空间、账号或资源边界。
- 标准化的密钥管理和工作负载身份方案。
- 从第一天开始设置资源配额和成本标签。

### 可观测性与可靠性层

可观测性不是上线后的补充仪表盘，而是团队判断交付是否真的变好、风险是否正在累积的基础。

平台应提供：

- 具备统一标签的日志、指标、链路追踪和事件。
- 围绕关键用户旅程定义 SLI 和 SLO。
- 与服务归属元数据绑定的告警路由。
- Runbook 模板和事故复盘流程。
- 部署标记，帮助团队关联发布与系统行为变化。

### 治理与安全层

治理不应停留在口口相传的经验里，而应沉淀为可重复执行的策略。平台可以把每个工作负载必须满足的最低要求编码化。

例如：

- 镜像来源证明和漏洞扫描。
- 通过 Policy as Code 管理运行时权限、网络访问和必填元数据。
- 基于角色或属性的最小权限访问控制。
- 对部署、基础设施变更和高权限操作进行审计。
- 针对存储、备份和密钥的数据分级规则。

## 3）设计黄金路径

黄金路径是从“新服务想法”到“安全进入生产”的最简单受支持流程。它应该回答这些问题：

1. 如何创建一个新的服务仓库？
2. 如何在本地运行它？
3. 如何申请标准依赖？
4. 如何部署到开发、预发和生产环境？
5. 如何查看日志、指标、链路追踪和告警？
6. 如何安全回滚？
7. 如何知道自己负责哪些资源？

黄金路径通常由模板、文档、CLI、内部门户和可复用流水线组件共同承载。它的核心指标是采用率：如果团队持续绕开它，说明平台可能太僵硬、太慢，或者文档不够清晰。

## 4）让环境策略可预测

环境策略是很多 DevOps 实践变复杂的地方。一个实用模型可以这样划分：

- **本地环境**：提供快速开发反馈，依赖尽量轻量。
- **开发环境**：与共享平台服务进行集成。
- **预发环境**：尽量接近生产，用于验证发布候选版本。
- **生产环境**：强调受控发布、监控和回滚。

不要把预发环境变成长期堆放实验资源的地方。如果团队需要临时验证，应提供与 Pull Request 或特性分支绑定的临时预览环境，并在不再需要时自动销毁。

## 5）度量平台成效

平台成功的标志，是交付速度和可靠性同时提升。可以关注这些指标：

- 从提交到生产的前置时间。
- 部署频率。
- 变更失败率。
- 平均恢复时间。
- 创建一个新服务所需时间。
- 使用黄金路径的服务占比。
- 告警噪音和每个服务的夜间唤醒次数。
- 按团队、服务和环境拆分的云成本。

指标应该反过来驱动平台团队的产品决策。如果创建服务需要几天，就优先完善模板和自助供给；如果部署变快后事故增加，就加强渐进式发布和自动验证。

## 6）渐进式落地

不要试图一次性替换所有工具和流程。更稳妥的推进方式是：

1. 选择一个有代表性的服务和一个愿意配合的产品团队。
2. 构建能让该服务安全进入生产的最小黄金路径。
3. 记录完整过程，并消除人工步骤。
4. 补齐可观测性、策略和回滚自动化。
5. 迁移第二、第三个服务，暴露缺失的抽象。
6. 把重复模式沉淀成可复用模块。
7. 发布采用指南和支持边界。

这种方式能让平台扎根真实需求，而不是停留在没人使用的架构图里。

## 7）常见反模式

需要警惕这些问题：

- **工具优先**：还没弄清工作流和责任边界，就先堆工具。
- **中心化瓶颈**：每次部署或基础设施变更都必须等待平台团队。
- **成本不可见**：环境和资源长期存在，却没有负责人和预算边界。
- **仪表盘表演**：图很多，但缺少可行动的 SLO 和告警。
- **安全只做最终检查**：团队已经按错误方式建设完毕，安全要求才被追加。
- **一刀切抽象**：平台隐藏了太多细节，反而阻碍合理的业务差异。

好的平台降低摩擦，但不会取消工程责任。

## 总结

DevOps 平台是一个长期演进的内部工程产品。它的职责是让安全路径变成最容易的路径：标准化流水线、可复用基础设施、可观测服务、编码化治理，以及清晰的归属关系。先从小范围真实场景开始，用指标验证效果，再通过实际采用不断演进，而不是追求纸面上的完整性。
